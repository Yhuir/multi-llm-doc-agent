# 预开发说明（pre.md）

版本：V1.1  
状态：可进入 Codex Desktop / Cursor / 人工开发  
适用范围：Multi-Agent 工程实施方案自动生成系统 V1

---

## 1. 目标

构建一个“上传技术要求 Word → 自动生成目录 → 用户交互修订 → 确认目录 → 分节点生成正文/图片/表格 → Word 排版导出”的多 Agent 系统。

V1 目标不是做“最强 AI 写作器”，而是做“可落地、可恢复、可审阅、可导出”的工程文档生产流水线。

---

## 2. 已确认的核心决策

### 2.1 输入格式
- 唯一正式支持：`.docx`
- `.doc` 不作为正式支持格式
- UI 和后端都应明确提示：请上传 `.docx`

### 2.2 最小生成单元
- 最小生成单元为 **三级/四级目录自适应**
- 默认先生成到三级
- 若三级节点不适合作为独立生成单元，则自动下钻到四级
- 一旦用户确认目录并开始正文生成，该任务的最小生成层级即固定，不允许中途改变

### 2.3 四级目录触发规则
系统按如下规则判断是否将三级节点下钻到四级：

满足任一条件即可触发：
1. 三级节点覆盖多个子系统/模块
2. 三级节点标题过于宽泛，无法稳定映射到单一施工主题
3. 预估正文不足以达到 1800 字硬下限
4. requirement 侧关联条款过多，直接写三级节点容易内容混杂

说明：本规则以“可独立生成、可独立校验、可独立排版”为判断原则，而不是为了层级更深而更深。

### 2.4 字数规则
- 最终正文硬下限：`>= 1800`
- 推荐范围：`1800–2200`
- 超过 `2200`：触发精简
- 精简优先删减空泛描述、重复描述，不删除关键参数、验收条款、测试步骤、风险控制、留痕要求

### 2.5 页数规则
- 每个最小生成节点版面目标：`2–4 页`
- 若节点包含 2–3 张图片或表格，可超过 2 页
- “2 页左右”仅作为参考目标，不作为硬验收项
- 以内容完整性优先，而不是强行压页数

### 2.6 图片策略
- 每个最小生成节点生成 `2–3` 张强关联图片
- V1 不追求“真的懂图”
- 仅做工程可用版：
  - 从正文中抽取实体
  - 生成受约束的图片 prompt
  - 用轻量的图文相关性校验做 pass/fail
  - fail 后自动重试
- 图片失败策略：**宽松模式**
  - 单图多次失败后标记 `NEED_MANUAL_CONFIRM`
  - 节点可继续排版
  - UI 明确提示该节点需要人工确认图片

### 2.7 防瞎编（事实可追溯）
系统必须加入 **Fact Grounding Agent**，要求生成内容“有据可查”，不能瞎编。

原则：
- 设备型号、数量、标准号、阈值、接口关系、验收依据等关键事实，必须能在 requirement 解析结果中找到来源，或者能明确标记为“工程通用补充项”
- 未找到来源的关键陈述，必须标记 unsupported
- unsupported 超过阈值时触发重写/修订

### 2.8 目录冻结规则
- 用户点击“确认目录 / 开始生成内容”后：
  - 当前目录版本冻结为 `toc_confirmed`
  - 生成中不允许再改目录
- 若需改目录：
  - 必须新建派生任务
  - 原任务保留不变

### 2.9 目录版本管理
- 目录版本按 `toc_v1 / toc_v2 / toc_v3...` 存储
- 用户可回滚到旧版本继续修改
- 回滚后，后续版本保留，不删除
- 差异对比采用 **树级 diff**
- 节点标识采用双 ID：
  - `node_uid`：跨版本稳定 ID
  - `node_id`：当前版本展示编号

### 2.10 表格规则
- 仅当系统判断某段内容适合表格表达时，才创建表格
- 所有生成表格统一使用模板中的 `BiddingTable` 样式
- 不允许为了“看起来专业”而滥用表格

### 2.11 运行架构
- V1 架构采用：`React 前端 + 后台 Worker`
- 不做“仅前端单体长任务”方案
- 状态持久化：SQLite
- 工件持久化：本地文件系统 `artifacts/{task_id}/...`

### 2.12 并发策略
V1 默认：
- 文本按节点串行
- 节点内图片可并发 2–3 张
- 不同节点不并发
- 优先追求稳定与可恢复，不优先追求极限速度

## 3. 产品边界

### 3.1 V1 In Scope
- `.docx` 全文解析
- requirement 结构化抽取
- 目录树生成
- 目录交互修订、版本管理、树级 diff
- 用户确认目录后分节点生成正文
- 生成表格（按需）
- 生成图片（2–3 张）
- 图文一致性校验与重试
- 字数补写与精简
- 事实可追溯校验
- 一致性校验
- 基于模板样式的 Word 排版
- 导出 `output.docx`
- 实时进度、实时日志、断点续跑

### 3.2 V1 Out of Scope
- 深度多模态图像语义理解
- 多用户协作编辑
- 在线 Word 所见即所得编辑器
- 自动从互联网补全缺失技术条款
- 节点级高并发生成
- 分布式任务队列

## 4. 业务流程

### 4.1 主流程
1. 新建任务 / 继续任务
2. 上传 `.docx`
3. Requirement Parser 解析原文
4. 生成 `requirement.json`
5. TOC Generator 生成 `toc_v1`
6. 用户在 AI 对话区提交目录修改意见
7. TOC Review Agent 生成 `toc_v2 / toc_v3...`
8. 用户点击“确认目录 / 开始生成内容”
9. 系统冻结目录为 `toc_confirmed`
10. Worker 按最小生成单元逐个执行：
   - 生成正文
   - 事实校验
   - 抽取实体
   - 生成图片 prompt
   - 出图
   - 图文相关性校验
   - 字数校验与补写/精简
   - 一致性校验
   - 进入排版
11. 全部节点完成后生成 Word
12. 用户下载 `output.docx`

### 4.2 中断恢复
- 任意时刻进程退出、机器重启、网络异常后，任务均可恢复
- 恢复原则：
  - 已完成阶段不重复执行
  - 进行中阶段根据心跳和状态决定回滚或重试
- UI 必须支持：
  - 查看未完成任务
  - 继续任务
  - 查看历史目录版本
  - 查看对话历史
  - 查看节点日志与指标

## 5. 核心非功能要求

### 5.1 可恢复性
- 必须以最小生成单元为 checkpoint 粒度
- 每阶段结束写入持久化状态
- 每阶段保存输入快照与输出工件路径

### 5.2 可追溯性
- 生成内容需可回溯到 requirement 侧来源
- 事件日志可审计
- 节点失败原因可定位

### 5.3 可解释性
UI 至少能解释：
- 当前跑到哪个节点
- 当前处于哪个阶段
- 为什么失败
- 重试了几次
- 事实校验 / 图文校验 / 字数校验是否通过

### 5.4 稳定性优先
V1 以稳定可用为优先级，高于性能与花哨效果。

## 6. Fact Grounding 规则

### 6.1 目标
防止以下问题：
- 编造设备型号
- 编造数量
- 编造标准号
- 编造验收阈值
- 编造接口关系
- 编造项目工期或里程碑

### 6.2 判断策略
将 node_text 中的关键事实拆成 claim，按类别校验：

- 设备/型号
- 数量/位置
- 参数/阈值
- 标准/规范
- 子系统关系
- 里程碑/工期
- 验收要求

每条 claim 标记：
- `SUPPORTED`
- `WEAKLY_SUPPORTED`
- `UNSUPPORTED`
- `GENERAL_ENGINEERING_KNOWLEDGE`

### 6.3 通过建议
- 关键 claim 不允许出现 `UNSUPPORTED`
- `grounded_ratio >= 0.70`
- 若不通过，则先触发 revise
- revise 后仍失败，则节点进入 `WAITING_MANUAL`

---

## 7. 一致性检查分类

### 7.1 实体一致性
检查：
- 设备名称
- 型号
- 数量
- 位置
- 接口定义

### 7.2 术语一致性
检查：
- 同一概念是否存在多个称呼
- 前后文是否术语混用
- 标题与正文术语是否一致

### 7.3 约束一致性
检查：
- 标准号
- 参数
- 阈值
- 项目工期
- 施工条件

### 7.4 引用一致性
检查：
- 图号
- 表号
- 章节号
- 节点引用
- 图片绑定锚点

---

## 8. 进度计算原则

### 8.1 总进度
总进度 = 基础阶段 + 节点生成阶段 + 导出阶段

建议权重：
- 解析与目录审阅前置阶段：20%
- 节点生成阶段：70%
- 排版与导出：10%

### 8.2 节点进度
单节点阶段权重建议：
- 正文生成：35%
- 图片生成：20%
- 图文校验：15%
- 字数控制：10%
- 事实校验：5%
- 一致性校验：5%
- 排版入队：10%

要求：
- UI 实时刷新
- 节点高亮
- 完成节点打勾
- 失败节点标红
- 需要人工确认节点标橙色

## 10. V1 验收基线

一个任务被认为“可交付完成”，至少满足：
1. 成功上传并解析 `.docx`
2. 成功生成 TOC 且支持版本修订
3. 目录确认后可按节点持续生成
4. 断点恢复可用
5. 绝大多数节点满足 `>=1800` 字
6. 失败图片不会阻塞整任务
7. 事实校验与一致性校验可输出结果
8. Word 样式沿用模板
9. 能输出 `output.docx`
10. UI 有可用的实时进度与日志
